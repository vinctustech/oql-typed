If you already have an OQL `.dm` data model, the `oql-typed-codegen` CLI generates a TypeScript schema from it. This is the fastest way to adopt `@vinctus/oql-typed` in an existing project — and the recommended way to keep your `.dm` file and your TypeScript schema in sync.

## The intended workflow

Treat the `.dm` file as the source of truth. Re-run codegen whenever it changes — the generated TypeScript imports `defineSchema`, `entity`, columns, and relations from `@vinctus/oql-typed` exactly as you'd write them by hand.

<Aside type="note">
The reverse direction — generating `.dm` from a TypeScript schema with `generateDM(...)` — is currently a stub that throws. It's being rebuilt for the schema-object API. For now, the `.dm` file is the source of truth; bootstrap or regenerate `schema.ts` from it.
</Aside>

## Command-line

<Steps>

1. **Point it at your `.dm` file and an output path.**

   ```bash
   npx oql-typed-codegen schema.dm src/schema.generated.ts
   ```

2. **Import the generated schema.**

   The generated file exports a single `schema` constant built with `defineSchema(...)`.

   ```typescript
   import { schema } from './schema.generated.js'
   import { typedOQL } from '@vinctus/oql-typed'

   const db = typedOQL(oql, schema)
   ```

3. **Start writing typed queries.**

   ```typescript
   import { eq } from '@vinctus/oql-typed'

   await db.user.where(eq(db.user.enabled, true)).many()
   ```

</Steps>

### Reading from stdin

Pipe `.dm` source in on stdin by passing `-` as the input path:

```bash
cat schema.dm | npx oql-typed-codegen -
```

When no output path is given, the generated TypeScript is printed to stdout — handy for `|`-chains or quick inspection.

## Programmatic

```typescript
import { parseDMAndGenerate } from '@vinctus/oql-typed'

const tsSource = parseDMAndGenerate(dmString)
```

Same output as the CLI — a TypeScript module that exports a `schema` built with `defineSchema(...)`.

For finer control, the two-stage form is also exported:

```typescript
import { parseDM, generateSchemaTS } from '@vinctus/oql-typed'

const parsed   = parseDM(dmString)
const tsSource = generateSchemaTS(parsed)
```

## What the generated file looks like

Running the codegen on a `.dm` like:

```oql
entity user (users) {
  *id: uuid
  email: text!
  account (account_id): account!
}

entity account (accounts) {
  *id: uuid
  name: text!
  users: [user]
}
```

produces:

```typescript
import { defineSchema, entity, manyToOne, oneToMany, text, uuid } from '@vinctus/oql-typed'

export const schema = defineSchema({
  user: entity('users', {
    id: uuid().primaryKey(),
    email: text(),
    account: manyToOne('account', { column: 'account_id' }),
  }),
  account: entity('accounts', {
    id: uuid().primaryKey(),
    name: text(),
    users: oneToMany('user'),
  }),
})
```

No `@ts-nocheck` header — the schema-object design preserves all type information under strict mode.

## Hand-authoring is fine too

The generator emits exactly the API you'd use by hand. You can write `defineSchema(...)` yourself and skip codegen entirely — useful when the schema is small, when you want strong control over inline type aliases (`type UserRole = ...`), or when you don't have a `.dm` to start from.

If you go that route, keep a `.dm` file alongside it for OQL itself to consume at startup. They have to stay in sync — there is no schema-object-to-`.dm` generator at the moment.
