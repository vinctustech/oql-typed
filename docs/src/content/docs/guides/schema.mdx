Your schema is the source of truth. Define it once in TypeScript with `defineSchema(...)` and both runtime OQL and compile-time types fall out of it.

## Shape of a schema

A schema is a single object passed to `defineSchema(...)`. Each entry maps an **entity name** (the key) to an entity definition produced by `entity(...)`. Relations reference other entities by **string literal**, so there are no circular-import problems.

```typescript
import { defineSchema, entity, uuid, text, boolean } from '@vinctus/oql-typed'

export const schema = defineSchema({
  category: entity('categories', {
    id:      uuid().primaryKey(),
    name:    text(),
    slug:    text(),
    enabled: boolean(),
  }),
})
```

`entity(...)` has two forms:

- **`entity(fields)`** â€” the entity name doubles as the table name.
- **`entity(tableName, fields)`** â€” use when your SQL table name differs from the entity name.

The **entity name** (used in `query(db, 'category')`, OQL strings, and relation targets) is the **key** in the `defineSchema({...})` object â€” not an argument to `entity()`.

<Aside type="note">
`defineSchema` is just an identity function with a `const` generic â€” it preserves literal types so TypeScript can infer everything (entity names, enum values, field shapes) without manual annotations. You can pass plain objects to `typedOQL(...)` too, but `defineSchema` is preferred.
</Aside>

## Column types and modifiers

### Built-in column constructors

```typescript
import {
  uuid, text, integer, bigint, float, decimal,
  boolean, timestamp, date, time, interval,
  json, textArray, integerArray, enumType,
} from '@vinctus/oql-typed'
```

| Constructor | TypeScript type | PostgreSQL type |
|-------------|-----------------|-----------------|
| `uuid()` | `string` | `uuid` |
| `text()` | `string` | `text` |
| `integer()` | `number` | `integer` |
| `bigint()` | `bigint` | `bigint` |
| `float()` | `number` | `real` / `double precision` |
| `decimal(precision?, scale?)` | `bigint` | `decimal(p,s)` |
| `boolean()` | `boolean` | `boolean` |
| `timestamp()` | `Date` | `timestamp` |
| `date()` | `Date` | `date` |
| `time()` | `string` | `time` |
| `interval()` | `string` | `interval` |
| `json<T = unknown>()` | `T` | `json` / `jsonb` |
| `textArray()` | `string[]` | `text[]` |
| `integerArray()` | `number[]` | `integer[]` |
| `enumType<T>(name, values)` | `T` | custom enum |

### Modifiers

Chain modifiers after a column constructor:

```typescript
user: entity('users', {
  id:          uuid().primaryKey(),                             // marks PK
  firstName:   text().column('first_name'),                     // renames column
  email:       text(),
  lastLoginAt: timestamp().column('last_login_at').nullable(),  // nullable + rename
})
```

| Modifier | Effect |
|----------|--------|
| `.primaryKey()` | Marks this column as the primary key. |
| `.nullable()` | Adds `\| null` to the inferred type; the column is allowed to be `NULL`. |
| `.column('alias')` | Uses a different database column name than the TypeScript field name. |

<Aside type="note">
`.nullable()` changes both the TypeScript type and the underlying DM definition. Mismatching nullability between your schema and the database is one of the most common bugs typed queries catch.
</Aside>

## Enums

```typescript
type PostStatus = 'DRAFT' | 'PUBLISHED' | 'ARCHIVED'

post: entity('posts', {
  id:     uuid().primaryKey(),
  title:  text(),
  status: enumType<PostStatus>('PostStatus', ['DRAFT', 'PUBLISHED', 'ARCHIVED']),
})
```

Operators reject invalid enum values at compile time:

```typescript
eq(db.post.status, 'PUBLISHED')   // âś“
eq(db.post.status, 'APPROVED')    // âś— type error
```

## Relations

Relations target other entities **by string literal** â€” the same name you use as the key in `defineSchema(...)`. No `() => ref` thunks, no circular-import worries.

| Constructor | Shape at the type level | OQL emits |
|-------------|-------------------------|-----------|
| `manyToOne(target, { column })` | One target row | LEFT JOIN |
| `oneToMany(target)` | `Target[]` | lateral subquery |
| `manyToMany(target, { junction })` | `Target[]` | lateral subquery with junction join |
| `oneToOne(target, { reference? })` | One target row | lateral subquery (single) |

### manyToOne â€” foreign key

```typescript
post: entity('posts', {
  id:       uuid().primaryKey(),
  title:    text(),
  author:   manyToOne('user',     { column: 'author_id' }),
  category: manyToOne('category', { column: 'category_id' }).nullable(),
}),
```

With `manyToOne`, field access builds **dotted paths** that keep their types:

```typescript
db.post.author             // RelationFieldRef
db.post.author.id          // FieldRef<string>
db.post.author.email       // FieldRef<string> (one-level hop)
db.post.category.id        // FieldRef<string | null> (propagates nullability)
```

### oneToMany â€” reverse FK

```typescript
user: entity('users', {
  id:    uuid().primaryKey(),
  posts: oneToMany('post'),
}),
```

### manyToMany â€” junction table

```typescript
post: entity('posts', {
  id:   uuid().primaryKey(),
  tags: manyToMany('tag', { junction: 'post_tags' }),
}),

tag: entity('tags', {
  id:    uuid().primaryKey(),
  name:  text(),
  posts: manyToMany('post', { junction: 'post_tags' }),
}),
```

### oneToOne

```typescript
user: entity('users', {
  id:      uuid().primaryKey(),
  profile: oneToOne('profile'),
}),

profile: entity('profiles', {
  id:      uuid().primaryKey(),
  user:    manyToOne('user', { column: 'user_id' }),
  bio:     text(),
  website: text().nullable(),
}),
```

The optional `{ reference }` option points at the field on the target that holds the FK back â€” useful when the owning side has a non-default name:

```typescript
user: entity('users', {
  id:      uuid().primaryKey(),
  vehicle: oneToOne('vehicle', { reference: 'driver' }).nullable(),
}),
```

### Self-referencing manyToOne

Self-references work out of the box because targets are strings:

```typescript
comment: entity('comments', {
  id:     uuid().primaryKey(),
  body:   text(),
  parent: manyToOne('comment', { column: 'parent_id' }).nullable(),
}),
```

### Two relations to the same target

Just give them different field names and column aliases:

```typescript
post: entity('posts', {
  id:         uuid().primaryKey(),
  title:      text(),
  author:     manyToOne('user', { column: 'author_id' }),
  reviewedBy: manyToOne('user', { column: 'reviewed_by_id' }).nullable(),
}),
```

## Wiring up the runtime

`typedOQL(oqlInstance, schema)` returns a typed `db` handle. The schema you pass is the same object you got from `defineSchema(...)`:

```typescript
import { typedOQL } from '@vinctus/oql-typed'
import { OQL_PG } from '@vinctus/oql-pg'
import { schema } from './schema.js'

// dm is the OQL data-model string (see the codegen guide for how to get it).
const oqlInstance = new OQL_PG(dm, host, port, database, username, password)

export const db = typedOQL(oqlInstance, schema)
```

`db.post.id`, `db.post.author.email`, `db.post.select(...)`, etc. all become available with full type inference.

## The full blog schema

Every column type, every relation kind, every modifier. The examples elsewhere in these docs draw from this schema.

```typescript
import {
  defineSchema, entity,
  uuid, text, integer, bigint, float, decimal,
  boolean, timestamp, date, time, interval,
  json, textArray, integerArray, enumType,
  manyToOne, oneToMany, manyToMany, oneToOne,
} from '@vinctus/oql-typed'

export type UserRole = 'ADMIN' | 'AUTHOR' | 'READER'
export type PostStatus = 'DRAFT' | 'PUBLISHED' | 'ARCHIVED'

export const schema = defineSchema({
  user: entity('users', {
    id:          uuid().primaryKey(),
    email:       text(),
    firstName:   text().column('first_name'),
    lastName:    text().column('last_name'),
    role:        enumType<UserRole>('UserRole', ['ADMIN', 'AUTHOR', 'READER']),
    enabled:     boolean(),
    rating:      float().nullable(),
    metadata:    json().nullable(),
    lastLoginAt: timestamp().column('last_login_at').nullable(),
    profile:     oneToOne('profile'),
    posts:       oneToMany('post'),
    comments:    oneToMany('comment'),
  }),

  profile: entity('profiles', {
    id:      uuid().primaryKey(),
    user:    manyToOne('user', { column: 'user_id' }),
    bio:     text(),
    website: text().nullable(),
  }),

  category: entity('categories', {
    id:   uuid().primaryKey(),
    name: text(),
    slug: text(),
  }),

  tag: entity('tags', {
    id:    uuid().primaryKey(),
    name:  text(),
    posts: manyToMany('post', { junction: 'post_tags' }),
  }),

  post: entity('posts', {
    id:             uuid().primaryKey(),
    title:          text(),
    slug:           text(),
    body:           text(),
    status:         enumType<PostStatus>('PostStatus', ['DRAFT', 'PUBLISHED', 'ARCHIVED']),
    featured:       boolean(),
    viewCount:      integer().column('view_count'),
    wordCount:      bigint().column('word_count'),
    readingTime:    decimal(4, 2).column('reading_time').nullable(),
    createdAt:      timestamp().column('created_at'),
    publishedAt:    timestamp().column('published_at').nullable(),
    publishedOn:    date().column('published_on').nullable(),
    publishTime:    time().column('publish_time').nullable(),
    editWindow:     interval().nullable(),
    keywords:       textArray(),
    visitHistogram: integerArray().column('visit_histogram'),
    metadata:       json().nullable(),
    author:         manyToOne('user',     { column: 'author_id' }),
    reviewedBy:     manyToOne('user',     { column: 'reviewed_by_id' }).nullable(),
    category:       manyToOne('category', { column: 'category_id' }).nullable(),
    comments:       oneToMany('comment'),
    tags:           manyToMany('tag', { junction: 'post_tags' }),
  }),

  comment: entity('comments', {
    id:        uuid().primaryKey(),
    body:      text(),
    createdAt: timestamp().column('created_at'),
    post:      manyToOne('post',    { column: 'post_id' }),
    author:    manyToOne('user',    { column: 'author_id' }),
    parent:    manyToOne('comment', { column: 'parent_id' }).nullable(),
  }),
})
```

What's exercised here:

- **All scalar column types**: uuid, text, integer, bigint, float, decimal, boolean, timestamp, date, time, interval, json, textArray, integerArray.
- **Enum columns** (`UserRole`, `PostStatus`) with compile-time-checked values.
- **All modifiers**: `.primaryKey()`, `.nullable()`, `.column()`.
- **Every relation kind**: `manyToOne`, `oneToMany`, `manyToMany`, `oneToOne`.
- **Two `manyToOne`s to the same target** (`post.author` and `post.reviewedBy`).
- **Self-referencing `manyToOne`** (`comment.parent`).
