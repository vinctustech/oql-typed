Most list endpoints have a shape like: always filter by `enabled = true`, and optionally filter by a search term, a role, a date range, etc. `queryBuilder(db, 'entity')` makes that pattern declarative.

## `.cond(value, filter)`

`queryBuilder` has everything `query` does, plus a `.cond(value, filter)` method that applies `filter` only when `value` is truthy.

<Tabs syncKey="cond">
  <TabItem label="TypeScript">
    ```typescript
    import { queryBuilder, eq, ilike, desc } from '@vinctus/oql-typed'

    const results = await queryBuilder(db, 'user')
      .select('id', 'firstName', 'role')
      .where(eq(db.user.enabled, true))
      .cond(role,   eq(db.user.role, role))                     // applied if role is truthy
      .cond(search, ilike(db.user.firstName, `%${search}%`))    // applied if search is truthy
      .orderBy(desc(db.user.lastLoginAt))
      .limit(size)
      .offset(page * size)
      .many()
    ```
  </TabItem>
  <TabItem label="No filters">
    ```oql
    user {id firstName role} [enabled = :p0]
      <lastLoginAt DESC>
      |50, 0|
    ```
  </TabItem>
  <TabItem label="Role filter only">
    ```oql
    user {id firstName role}
      [enabled = :p0 AND role = :p1]
      <lastLoginAt DESC>
      |50, 0|
    ```
  </TabItem>
  <TabItem label="Both filters">
    ```oql
    user {id firstName role}
      [enabled = :p0 AND role = :p1 AND firstName ILIKE :p2]
      <lastLoginAt DESC>
      |50, 0|
    ```
  </TabItem>
</Tabs>

Falsy values skip the filter entirely — no empty `WHERE` clause, no wasted parameter slots.

## Mixing with `.findBy()` / `.findIn()`

`queryBuilder` exposes the same `.findBy(col, v)` and `.findIn(col, vs)` shortcuts as `query()`, and they compose naturally with `.where()` / `.cond()` — all filter contributions are ANDed together in the order they were added.

```typescript
queryBuilder(db, 'post')
  .select('id', 'title')
  .findBy(db.post.status, 'PUBLISHED')          // mandatory eq
  .cond(authorId, eq(db.post.author, authorId)) // optional eq
  .findIn(db.post.tag, ['typescript', 'oql'])   // mandatory IN
  .many()
```

```oql
post {id title}
  [status = :p0 AND author.id = :p1 AND tag.id IN :p2]
```

`.findById()` is **not** available on `queryBuilder` — it auto-terminates with `.one()`, which conflicts with the conditional-builder shape. Use `.findBy(db.entity.id, id).one()` if you need that.

## List + total count

For paginated endpoints that return both a page of rows and a total count, build the query once and run two terminals:

```typescript
const base = queryBuilder(db, 'post')
  .select('id', 'title', { author: ['firstName'] })
  .where(eq(db.post.status, 'PUBLISHED'))
  .cond(search, ilike(db.post.title, `%${search}%`))
  .cond(categoryId, eq(db.post.category, categoryId))

const [items, total] = await Promise.all([
  base.orderBy(desc(db.post.publishedAt)).limit(size).offset(page * size).many(),
  base.count(),
])
```

<Aside type="note">
`queryBuilder` returns a reusable builder — `.many()`, `.count()`, and `.one()` don't mutate the builder, so you can call several terminals on the same instance.
</Aside>

## Realistic endpoint

A typical REST handler combining query-string parsing, conditional filters, and pagination:

```typescript
// Express / similar
app.get('/posts', async (req, res) => {
  const { search, status, categoryId, page = 0, size = 25 } = req.query

  const base = queryBuilder(db, 'post')
    .select('id', 'title', 'slug', 'publishedAt', { author: ['id', 'firstName'] })
    .where(eq(db.post.status, 'PUBLISHED'))
    .cond(search, ilike(db.post.title, `%${search}%`))
    .cond(status, eq(db.post.status, status))
    .cond(categoryId, eq(db.post.category, categoryId))

  const [items, total] = await Promise.all([
    base
      .orderBy(desc(db.post.publishedAt))
      .limit(Number(size))
      .offset(Number(page) * Number(size))
      .many(),
    base.count(),
  ])

  res.json({ items, total })
})
```
