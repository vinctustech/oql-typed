When a query needs SQL functions, subqueries, or aliased projections, oql-typed has expression helpers that plug into the type system. The escape hatch for anything unmodelled is `raw()`.

## `fn(name, ...args)` — generic function call

For arbitrary SQL functions without a typed wrapper:

<Tabs syncKey="fn">
  <TabItem label="Function in a filter">
    ```typescript
    import { fn, raw, ilike } from '@vinctus/oql-typed'

    ilike(fn('concat', db.user.firstName, raw("' '"), db.user.lastName), '%john%')
    ```

    ```oql
    concat(firstName, ' ', lastName) ILIKE :p0
    ```
  </TabItem>
  <TabItem label="Function in a projection">
    ```typescript
    import { query, fn, alias } from '@vinctus/oql-typed'

    query(db, 'user').select(
      'id',
      alias('fullName', fn('concat', db.user.firstName, raw("' '"), db.user.lastName)),
    )
    ```

    ```oql
    user {id fullName: concat(firstName, ' ', lastName)}
    ```
  </TabItem>
</Tabs>

### Typed function wrappers

Common SQL functions have dedicated wrappers that infer the result type:

```typescript
import {
  lower, upper, trim, length, concat, concatOp, coalesce,
  count, sum, avg, min, max,
} from '@vinctus/oql-typed'
```

| Wrapper | Result type | Notes |
|---------|-------------|-------|
| `lower(f)`, `upper(f)`, `trim(f)` | propagates `\| null` from input | NULL on NULL input |
| `length(f)` | `number` (`\| null` if input nullable) | NULL on NULL input |
| `concat(...args)` | `string` | PG's `concat()` — `STABLE`, **not indexable** |
| `concatOp(...args)` | `string \| null` | Emits `(a \|\| b \|\| ...)` — `IMMUTABLE`, **indexable** |
| `coalesce(...args)` | inferred (see below) | First non-null wins |
| `count(f)`, `count('*')` | `number` | Always non-null |
| `sum(f)`, `avg(f)` | `number \| null` | NULL on empty set |
| `min(f)`, `max(f)` | `T \| null` | NULL on empty set |

```typescript
ilike(lower(db.user.email), `%${term.toLowerCase()}%`)

db.comment.select(alias('total',       count('*')))
db.comment.select(alias('replyCount',  count(db.comment.id)))
db.post.select(alias('totalViews',     sum(db.post.viewCount)))
db.post.select(alias('avgReading',     avg(db.post.readingTime)))
```

#### `concat` vs `concatOp` — pick the right one

PostgreSQL's `concat(...)` function is marked `STABLE`, so an expression index built on it can't be used by the planner. The `||` operator is `IMMUTABLE` and **is** indexable — but it propagates `NULL` (if any operand is `NULL`, the result is `NULL`).

Use `concatOp(...)` when you need the expression to match a PG expression index:

```sql
CREATE INDEX user_full_name_idx ON users ((first_name || ' ' || last_name));
```

```typescript
import { concatOp, ilike, raw } from '@vinctus/oql-typed'

// Hits user_full_name_idx
ilike(concatOp(db.user.firstName, raw("' '"), db.user.lastName), '%john%')
```

Use `concat(...)` when you want NULL-safe behavior and don't need an index:

```typescript
import { concat } from '@vinctus/oql-typed'

concat(db.user.firstName, ' ', db.user.lastName)   // 'John ' even if lastName is NULL
```

#### `coalesce(...)` nullability

`coalesce` is overloaded: if **any** argument is non-null at the type level, the result is non-null. This matches the SQL semantics — the chain only collapses to NULL if every argument is NULL.

```typescript
coalesce(db.post.publishedAt, db.post.createdAt)   // Date (createdAt is non-null)
coalesce(db.post.publishedAt, null)                // Date | null (all args nullable)
```

## `raw(text)` — literal OQL escape hatch

When a construct isn't modelled, emit it literally.

```typescript
import { raw } from '@vinctus/oql-typed'

query(db, 'post').select('id', raw('totalViews: sum(viewCount)'))
```

```oql
post {id totalViews: sum(viewCount)}
```

<Aside type="caution">
`raw()` bypasses the type system. Prefer typed helpers whenever possible; reach for `raw()` only for things oql-typed doesn't yet model.
</Aside>

## `ref(m2o)` — the `&` reference operator

OQL's `&field` operator returns the foreign-key **column value itself**, rather than joining to the target entity. Useful when you only need to test the FK (e.g. for null-checks) without loading the related row.

```typescript
import { ref, isNull } from '@vinctus/oql-typed'

isNull(ref(db.comment.parent))          // top-level comments
```

```oql
&parent IS NULL
```

## `subquery<T>(relation, projection, filter?)` — parenthesized subquery

Use a scalar subquery as a value in another expression. The generic `<T>` tells TypeScript what type the subquery resolves to:

<Tabs syncKey="subq">
  <TabItem label="Posts with no comments">
    ```typescript
    import { subquery, eq } from '@vinctus/oql-typed'

    query(db, 'post')
      .select('id', 'title')
      .where(eq(subquery<number>(db.post.comments, ['count(*)']), 0))
    ```

    ```oql
    post {id title} [(comments {count(*)}) = :p0]
    ```
  </TabItem>
  <TabItem label="Users with more than 10 posts">
    ```typescript
    import { subquery, gt } from '@vinctus/oql-typed'

    query(db, 'user')
      .select('id', 'firstName')
      .where(gt(subquery<number>(db.user.posts, ['count(*)']), 10))
    ```

    ```oql
    user {id firstName} [(posts {count(*)}) > :p0]
    ```
  </TabItem>
</Tabs>

## `alias(label, expr)` — aliased projection

`alias('label', expr)` emits `label: expression` in the projection. The alias label and inferred type both flow into the result type.

```typescript
import { alias } from '@vinctus/oql-typed'

const rows = await query(db, 'comment')
  .select('id', alias('parentId', db.comment.parent.id))
  .many()
// → { id: string, parentId: string | null }[]
```

```oql
comment {id parentId: parent.id}
```

## `aliasedRelation(label, relationName, spec)` — aliased sub-collection

The same idea, but for a nested relation. The relation is named by **string literal** (matching the field name on the entity). The outer alias label and the typed inner projection both end up in the result type.

<Tabs syncKey="arel">
  <TabItem label="TypeScript">
    ```typescript
    import { aliasedRelation, eq } from '@vinctus/oql-typed'

    db.post.select(
      'id',
      'title',
      aliasedRelation('topComments', 'comments', {
        fields: ['id', 'body'],
        where: eq(db.comment.author.role, 'ADMIN'),
      }),
    )
    // → {
    //     id: string,
    //     title: string,
    //     topComments: { id: string, body: string }[],
    //   }[]
    ```
  </TabItem>
  <TabItem label="Generated OQL">
    ```oql
    post {
      id
      title
      topComments: comments {id body} [author.role = :p0]
    }
    ```
  </TabItem>
</Tabs>

For aliased relations whose row shape can't be inferred (e.g. only `raw(...)` entries), pass an explicit `Shape` type argument:

```typescript
aliasedRelation<{ count: number }>('passengers', 'trips', {
  fields: [raw('count: sum(seats)')],
})
```
